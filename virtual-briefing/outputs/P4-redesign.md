## 二、9个技术的叙事角色重新定位

| 技术 | RL后训练提出的新要求 → 业界回应 |
|------|-------------------------------|
| **PRIME-RL** | 异步架构下训推必须解耦才能放大规模 → PRIME-RL用全异步编排让rollout与update完全并行，是**训练系统层对规模化需求的回应** |
| **SkyRL** | 异步数据有保鲜期，过期数据毒害策略 → SkyRL用staleness过滤机制丢弃过时rollout，是**框架层对数据新鲜度需求的回应** |
| **DAPO** | 完全同步不现实，算法必须容忍一定off-policy → DAPO用解耦裁剪放宽clip范围，是**算法层对异步容忍需求的回应** |
| **Dynamo** | Agent多轮对话要求推理集群感知会话状态来路由 → Dynamo用KV-aware调度把请求路由到持有该会话KV Cache的节点，是**调度层对有状态路由需求的回应** |
| **LMCache** | Agent长程对话的KV Cache反复生成是巨大浪费 → LMCache用跨请求KV Cache池化和复用，是**缓存层对状态复用需求的回应** |
| **TraCT** | KV Cache爆炸性增长对显存形成硬约束 → TraCT用注意力头压缩在硬件层减少KV存储开销，是**硬件层对显存压力的回应** |
| **SALT** | 长trajectory中奖励信号稀疏，不知道哪一步出了问题 → SALT用时序差分做step级归因，是**归因层对信用分配需求的回应** |
| **SWE-PRM** | 终局reward太粗，无法指导中间步骤改进 → SWE-PRM训练过程奖励模型对每步打分，是**建模层对过程监督需求的回应** |
| **HiPER** | 大规模探索中agent容易陷入reward hacking → HiPER用分层探索策略约束搜索空间，是**策略层对安全探索需求的回应** |

## 三、P4演讲话术（60-90秒）

> "P3我们推导出三大挑战。现在关键问题是：**这些挑战落在谁头上？**
>
> 看上面这张表。新鲜度困境，落在训练系统层——你必须重新设计训推交互的异步架构，还要在算法层容忍off-policy。有状态难题，落在推理系统层——推理引擎要从无状态调度升级到有状态会话管理，KV Cache要从用完即弃变成池化复用。奖励失控，落在奖励评估层——需要精细化归因和过程奖励建模。
>
> 这就是我想传达的核心判断：**Agent RL后训练不只是调算法，它是一个全栈问题。** 每一层都被提出了新的要求。
>
> 下面九个技术框就是业界对这些新要求的回应。比如Dynamo不是因为它调度做得好我们才关注它——而是因为Agent RL训练的rollout阶段要求推理集群具备KV-aware路由能力，Dynamo是NVIDIA给出的回应。每个技术都是同样的逻辑：**RL后训练提出需求，技术是回应。**
>
> 这对我们计算部门意味着什么？意味着我们不能只看算法论文——推理引擎、调度系统、缓存架构全都要跟着变。接下来我们逐层展开。"
