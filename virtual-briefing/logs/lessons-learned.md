# 教训与经验总结

## 失败点

### 1. 技术发散——做加法停不下来
- 原始PPT 9个技术 → v4替换8个变成6个 → 拉回后变成15+个
- 每轮虚拟汇报领导提一个新维度就加进去，没有做减法
- 根因：把"全面覆盖"当成目标，忘了洞察的价值在于聚焦判断

### 2. 故事线丢失——叙事被技术淹没
- 原始PPT叙事线清晰：Agent RL三大挑战 × 三层解法 → 全栈协同
- 迭代中加了harness迁移线、横切约束、差异化页、竞争定位页……主线被稀释
- 听众听完记不住核心观点

### 3. 过度追求"前沿"——变成技术综述
- 把2025技术全部替换为2026技术，看起来"更新"但丢了经过验证的成熟方案
- DAPO/Dynamo/LMCache等在2026年不但没过时反而更成熟了，不该删
- "前沿洞察"不等于"只讲最新的"，成熟技术的产业验证同样是洞察

### 4. 虚拟汇报变成了"加需求"的过程
- 领导的每个质疑都试图在PPT中解决，导致PPT越来越厚
- 正确做法：质疑→判断是否影响主线→影响则改，不影响则记录到parking-lot
- 不是所有反馈都要直接体现在PPT中

### 5. 专家讨论过度发散
- 三位专家（RL/架构/Agent系统）各自带入自己领域的视角，每个都要求"加进去"
- 缺少一个"做减法"的角色——主持人应该砍掉偏离主线的内容，而不是都采纳

## 经验

### 1. 洞察PPT的核心是"一条线"
- 能用一句话说清楚的主题
- 每页只服务这一条线
- 技术是论据不是主角

### 2. 减法比加法重要
- 每次修改先问"删掉它主线会断吗？"——不会断就删
- 技术数量硬上限（9-10个），超过就必须砍

### 3. 原始PPT的骨架是对的
- 三大挑战 × 三层解法 = 9个技术的结构清晰、对称、易记
- 应该在这个骨架上做精准修正，不是推翻重来

### 4. 虚拟汇报的反馈要分级处理
- P0：主线逻辑断裂、核心数据错误 → 必须改PPT
- P1：重要但不影响主线 → 准备口头回应，不改PPT
- P2：发散议题 → 记parking-lot，明确不在本次PPT范围

### 5. 专家讨论要有"裁剪者"
- 主持人的首要职责不是"让每个人都满意"，是"守住主线"
- 对专家说"你的观点对但不在本次范围"是必要的

## 重新开始的原则

1. **回到原始PPT的骨架**：三大挑战×三层解法的结构
2. **只做精准修正**：逻辑缺环补上、过时数据更新、不准确表述修正
3. **技术选择标准**：原始9个技术逐一评估，过时的替换为2026对应版本，不过时的保留
4. **不加新结构**：不加横切约束页、不加差异化页、不加竞争定位页——这些内容如果需要就融入现有页面
5. **硬上限**：12页以内、9-10个技术以内
6. **我们的技术**：融入每页底部"对我们的含义"一行，不单独成页
