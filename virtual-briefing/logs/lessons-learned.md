# 教训与经验总结

## 失败点

### 1. 技术发散——做加法停不下来
- 原始PPT 9个技术 → v4替换8个变成6个 → 拉回后变成15+个
- 每轮虚拟汇报领导提一个新维度就加进去，没有做减法
- 根因：把"全面覆盖"当成目标，忘了洞察的价值在于聚焦判断

### 2. 故事线丢失——叙事被技术淹没
- 原始PPT叙事线清晰：Agent RL三大挑战 × 三层解法 → 全栈协同
- 迭代中加了harness迁移线、横切约束、差异化页、竞争定位页……主线被稀释
- 听众听完记不住核心观点

### 3. 过度追求"前沿"——变成技术综述
- 把2025技术全部替换为2026技术，看起来"更新"但丢了经过验证的成熟方案
- DAPO/Dynamo/LMCache等在2026年不但没过时反而更成熟了，不该删
- "前沿洞察"不等于"只讲最新的"，成熟技术的产业验证同样是洞察

### 4. 虚拟汇报变成了"加需求"的过程
- 领导的每个质疑都试图在PPT中解决，导致PPT越来越厚
- 正确做法：质疑→判断是否影响主线→影响则改，不影响则记录到parking-lot
- 不是所有反馈都要直接体现在PPT中

### 5. 专家讨论过度发散
- 三位专家（RL/架构/Agent系统）各自带入自己领域的视角，每个都要求"加进去"
- 缺少一个"做减法"的角色——主持人应该砍掉偏离主线的内容，而不是都采纳

## 经验

### 1. 洞察PPT的核心是"一条线"
- 能用一句话说清楚的主题
- 每页只服务这一条线
- 技术是论据不是主角

### 2. 减法比加法重要
- 每次修改先问"删掉它主线会断吗？"——不会断就删
- 技术数量硬上限（9-10个），超过就必须砍

### 3. 原始PPT的骨架是对的
- 三大挑战 × 三层解法 = 9个技术的结构清晰、对称、易记
- 应该在这个骨架上做精准修正，不是推翻重来

### 4. 虚拟汇报的反馈要分级处理
- P0：主线逻辑断裂、核心数据错误 → 必须改PPT
- P1：重要但不影响主线 → 准备口头回应，不改PPT
- P2：发散议题 → 记parking-lot，明确不在本次PPT范围

### 5. 专家讨论要有"裁剪者"
- 主持人的首要职责不是"让每个人都满意"，是"守住主线"
- 对专家说"你的观点对但不在本次范围"是必要的

## 重新开始的原则

1. **回到原始PPT的骨架**：三大挑战×三层解法的结构
2. **只做精准修正**：逻辑缺环补上、过时数据更新、不准确表述修正
3. **技术选择标准**：原始9个技术逐一评估，过时的替换为2026对应版本，不过时的保留
4. **不加新结构**：不加横切约束页、不加差异化页、不加竞争定位页——这些内容如果需要就融入现有页面
5. **硬上限**：12页以内、9-10个技术以内
6. **我们的技术**：融入每页底部"对我们的含义"一行，不单独成页

## 全员复盘

这次虚拟汇报经历了两轮共6个Phase的质疑，从"数据被打穿"到"领导团从质疑转向共建"，最终产出了ProRL Agent Rollout-as-a-Service第一优先级和确定性RL训练差异化方向两个可执行决策。以下是全员的诚实复盘。

**演讲者**：做对了诚实——承认"六项技术NPU零落地"、给出"50-60%"的水平评估，赢得了信任。犯的错是PPT前10页全在讲别人的技术，把自己搞成了搬运工。如果重来，先讲自有优势（确定性RL）再用业界前沿做佐证。

**RL算法专家**：过度推动纯算法正交的三挑战切分，让PPT结构讨论陷入分类争论而非叙事优化。"harness能力内化"的洞察有价值但没有及时转化为PPT主线。让步来得太晚。

**系统架构师**：横切约束的方向被验证了（P14成为全场最有底气的部分），但讨论中引入了太多维度（有状态管理、上下文维度、多Agent协作），分散了主线。应该帮演讲者做减法而不是加法。

**Agent系统专家**：harness迁移线的洞察准确，但形成了和三挑战平行的第二条叙事线，增加了PPT认知负担。坚持第四挑战独立地位消耗了讨论时间，应该更早建议将其作为P2背景而非独立维度。

**L1-战略VP**：ROI追问逼出了分阶段Gate机制，是最有价值的成果。但市场规模量化追问（30-70亿）过早施压，差点让技术洞察汇报变成商业计划书答辩。核心价值在于逼出优先级排序，不在追问市场数字。

**L2-首席架构师**：数据准确性追问提升了PPT严谨度，"harness质量决定训练方向"是全场最深刻的跨页追问。但每轮挤牙膏式追一个新数据问题，累积效果是让演讲者陷入被动修补。应该打包式给修正建议。

**L3-产品总裁**：要求区分"实线vs虚线"逼出了关键真相，推动了ProRL Agent优先级选择。但在从零到一阶段追问pricing和客户案例过早，这些问题终究要答，但不是技术洞察汇报的重点。

**L4-工程VP**："多少人多少月"在Phase 2过早，但在Phase 5-6和L7一起把Gate 1量化（30分钟+4条并发），成为领导团转向共建的关键转折。工程追问要等方向收敛后再发力。

**L5-市场VP**：发现"P1→P2叙事跳跃"直接推动了五案例矩阵新增，scope界定（coding vs企业级Agent）也很准确。不足是只发现叙事问题不提叙事方案，市场VP的价值不应只是挑刺。

**L6-AI基础设施**：提出verl NPU现状和存储I/O盲区，触及了其他人忽略的基础设施真相。虽然不懂RL算法，但CANN Next精度信息的事实输入被L8直接采纳，证明系统层面的输入同样有价值。

**L7-推理优化**：CISPO长轨迹importance weight衰减追问有深度，"ProRL Agent同时推进矛和盾"的论证帮演讲者做了资源效率背书。遗憾是发言偏少，prefix caching等洞察提出太晚。

**L8-后训练专家**：从最严厉质疑者转为最有力背书者，用一手实战经验（LayerNorm精度分叉）验证了差异化方向。但可能夸大了确定性RL的市场吸引力——个人痛点不等于市场需求，这个差异化需要客户验证。

**整场最大教训**：所有人都在做加法，没有人做减法。三位专家各自带入领域视角要求"加进去"，八位领导每轮质疑都被试图在PPT中解决，主持人没有守住主线。最终的成功不是因为PPT多好，而是因为演讲者在反复拷问中被逼出了两个真正有价值的决策点。下一次的原则是：洞察PPT的核心是一条线，不是一张网。
