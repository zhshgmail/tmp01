### P2 叙事逻辑（演讲者参考）

**上半页（三阶段演进 + 排除法）**：
后训练技术经历三代演进。第一代 SFT/RLHF（2023）奠定基础，但天花板是"能教模仿不能教决策"——Agent 状态空间太大，演示覆盖不了。第二代 DPO/KTO（2024）提升效率，但天花板是"能教对齐不能教交互"——Agent 的奖励来自环境验证，不来自人类偏好。第三代 Agentic RL（2025-2026）是唯一能系统性提升 Agent 在复杂环境中表现的路径：环境反馈替代人类标注，覆盖无限状态，从错误中学习。

三个维度同步佐证：后训练算力从<5%到40%+，代表模型从GPT-4升级到GPT-5.4/Claude Opus 4.6，场景从3-5种扩展到50+种。

**下半页（飞轮+巨头验证）**：
Agent 能力飞轮（模型更强→做事→轨迹→反馈→更新）的每个环节，都有巨头在工程化投入。不重复P1的事实列表，而是映射到飞轮节点上：Anthropic投环境、OpenAI开放反馈、Google做全流程训练。三家路线不同，但共识一致：RL后训练是关键。

**与 P1/P3 的边界**：
- P1 = 事实（谁做了什么产品、谁投了多少钱）
- P2 = 论证（为什么必须是 Agentic RL？演进全景+排除法+巨头验证）
- P3 = 推导（Agentic RL 和 LLM RL 的结构差异→架构→挑战）

## 第10页 | 技术②-1：NVIDIA Dynamo

（布局同第6页模板：左侧问题+示意图，右侧方案要点，底部关键逻辑+启示）

**问题**：Agent 多轮对话中大量 prefix 可复用，但标准负载均衡不知道哪个节点有哪段 KV cache，导致重复计算。

**方案**：
- NVIDIA GTC 2025 发布，开源推理框架
- KV-aware Router：Radix Tree 追踪全集群 KV 分布，计算 cache overlap score
- NIXL：点对点 KV cache 传输库
- KV Block Manager：多层级内存管理（GPU VRAM → CPU RAM → SSD）
- Prefill-Decode 分离

**关键逻辑**：推理调度从"哪个节点空闲"变成"哪个节点已有我需要的 KV"

**启示**：NPU 推理引擎需要支持 KV cache 元数据暴露和跨节点迁移能力

## 第11页 | 技术②-2：LMCache

（布局同第6页模板）

**问题**：每个推理引擎独立维护 KV cache，同一 prefix 被多个引擎重复计算。Agent 场景 95%+ context 可复用。

**方案**：
- 开源项目，集成 vLLM、SGLang、KServe
- P2P 共享：引擎间直接传输 KV cache
- Cache Offloading：GPU VRAM → CPU RAM 多级卸载
- Connector 接口：标准化 KV cache 存取 API
- 实测：多轮 QA 15x 吞吐提升，TTFT 降低 54.7%

**关键逻辑**：KV cache 不应是引擎私有资源，而应是集群级的共享池

**启示**：NPU 推理软件栈需支持标准化 KV cache 存取接口和高效跨节点传输

## 第12页 | 技术②-3：TraCT + CXL

（布局同第6页模板）

**问题**：大规模场景下 NIC 成为 KV 传输瓶颈。disaggregated serving 中 KV 传输占 TTFT 大头。

**方案**：
- 2025.12 学术论文，rack-scale LLM serving
- CXL 共享内存池：GPU 通过 CXL load/store 直接访问共享内存，绕过 NIC
- 两阶段同步：粗粒度 CXL DMA + 细粒度 load/store
- Prefix-aware 路由
- 实测：TTFT 降低 9.8x，P99 延迟降低 6.2x

**关键逻辑**：软件层优化到极限后，硬件互联升级才能打破天花板

**启示**：NPU 互联架构（HCCS 等）天然适合做 KV cache 跨节点共享，是硬件差异化机会
