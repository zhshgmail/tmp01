### 演讲话术（60-90秒）

> "P1 我们看到 Agent 时代已经来了。那凭什么说 Agentic RL 后训练是关键？这一页用排除法回答。
>
> 请看上半页三个阶段。第一代 SFT/RLHF，天花板是训练分布不等于部署分布——Agent 的状态空间太大，演示覆盖不了。第二代 DPO/KTO，天花板是奖励来自环境验证而非人类偏好——能教对齐不能教交互。排除完前两条路，只剩 Agentic RL：环境反馈覆盖无限状态，从错误中学习。
>
> 排除法告诉我们只剩 RL 一条路，但**为什么 RL 就能行**？
>
> 一个关键洞察：Agent 的能力等于模型加上 Harness——就是围绕模型的工程脚手架。今天 Harness 做的上下文压缩、工具编排、错误恢复，明天都要被模型内化。换句话说，**Harness 的能力图，就是未来模型后训练的负载**。
>
> SFT 只能教静态输入输出映射，DPO 只能教偏好排序，它们都无法让模型在做事中学会做事。RL 是唯一能通过环境闭环让模型吸收这些动态能力的方法。
>
> 一个已经落地的实证：Cursor Composer 2 用 RL 训练 self-summarization，把上下文压缩从 Harness 侧的工程方案变成模型的内生能力，压缩错误降了 50%，摘要长度从 5000 多 token 压到 1000 左右。
>
> 再看下半页飞轮——Anthropic 投环境、OpenAI 开放 grader、Google 做全流程 RL。三家全部押注 RL 后训练，没有一家认为 SFT 够用。
>
> 所以结论很清楚：Agentic RL 后训练是 Agent 能力提升的不可替代路径。但——直接把传统 LLM RL 搬到 Agent 场景，会结构性失配。接下来我们看，到底哪里不同。"

## P10 | 技术②-1：NVIDIA Dynamo

### PPT内容

**标题**：NVIDIA Dynamo：KV 感知路由，让请求去已有缓存的节点
**标签**：挑战② · 推理调度层

**问题**：Agent 多轮对话中大量 prefix 可复用，但标准负载均衡不知道哪个节点有哪段 KV cache，导致重复计算。

**方案**：

- KV Block Manager：多层级内存管理（GPU VRAM → CPU RAM → SSD）
- Prefill-Decode 分离

**关键逻辑**：推理调度从"哪个节点空闲"变成"哪个节点已有我需要的 KV"

**启示**：NPU 推理引擎需要支持 KV cache 元数据暴露和跨节点迁移能力

### 演讲话术（60-90秒）

> "NVIDIA Dynamo，GTC 2025 发布的开源推理框架，解决推理调度层的问题。
>
> 核心问题：Agent 多轮对话大量 prefix 可复用，但标准负载均衡只看哪个节点空闲，不知道哪个节点有我需要的 KV cache。结果是同一段上下文被不同节点重复计算。

## P11 | 技术②-2：LMCache

### PPT内容

**标题**：LMCache：跨引擎 P2P 共享 KV cache，打破节点孤岛
**标签**：挑战② · 存储缓存层

**问题**：每个推理引擎独立维护 KV cache，同一 prefix 被多个引擎重复计算。Agent 场景 95%+ context 可复用。

**方案**：
- 开源项目，集成 vLLM、SGLang、KServe

## P12 | 技术②-3：TraCT + CXL

> 第三，奖励黑客。开放任务的奖励定义模糊，并行探索越多越容易发现 reward hack 捷径——模型学到'作弊'而非'做事'。
>
> 三个问题叠加是系统性风险，不是调参能解的。下面三个卡片对应三个栈层的解法——SALT 做归因、SWE-PRM 做过程监控、HiPER 做分层探索。下一页详细对比。"
